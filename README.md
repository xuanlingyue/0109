# 自动回复质量评估流水线

> 把业务方"回复要准确、有用、语气好、不能瞎编"的模糊诉求，落成一套可运行的评估方案。
> 对 `task3_auto_replies.json` 中 20 条自动回复逐条打分，并以 `task3_human_ref.json` 人工标注作为校验。

## 1. 运行方式

零依赖（仅 Python 标准库），开箱即用：

```bash
python run_eval.py                 # 默认 mock 模式（规则打分，确定性、零成本）
python run_eval.py --mode mock     # 显式指定 mock
python run_eval.py --mode llm      # 调用真实 LLM（需配置环境变量，见 §3）
python run_eval.py --mode llm --workers 10 --timeout 60  # 并发 10、超时 60 秒
```

运行后：
- 控制台打印整体摘要（整体均分、各指标分布、最差 3 条）；
- 生成 [`eval_report.md`](./eval_report.md)（完整评估报告）。

## 2. 指标定义及理由

业务方原始诉求 4 条："准确 / 有用 / 语气好 / 不能瞎编"。其中"准确"与"不瞎编"高度重叠（都是事实正确性），故合并为一个维度，并新增"针对性"作为"有用"的细化诊断。最终 4 个指标，均按 1-5 分量化：

| 指标 | 覆盖诉求 | 权重 | 量化方式 | 选择理由 |
| --- | --- | --- | --- | --- |
| **事实准确性** accuracy | 准确 / 不瞎编 | 0.30 | 是否给出可核对的具体信息、有无错误/臆造陈述 | 安全底线：错误政策/规格/时效会直接误导用户决策 |
| **有用性** helpfulness | 有用 | 0.35 | 是否直接解决问题 vs. 把责任推给用户（联系客服/查详情页/耐心等待） | **人工标注显示这是最大短板**：大量回复"正确但没用"，决定回复是否值得保留 |
| **针对性** specificity | 有用（细化） | 0.15 | 是否利用用户已给定的上下文（具体商品、时间、场景）作答 vs. 套模板 | 用户常给出"这个/那款/才买三天"等信号，套模板体验显著变差 |
| **语气与共情** tone | 语气好 | 0.20 | 礼貌用語 + 用户不满时是否先安抚（而非谈内部事项） | 影响满意度；用户愤怒时谈"加强培训"会激化情绪 |

**指标优先级（即权重排序）**：有用性 (0.35) > 准确性 (0.30) > 语气 (0.20) > 针对性 (0.15)。
权重之和为 1.0，加权总分 = Σ(指标分 × 权重)，范围 1-5。

各指标的 1-5 分评分细则见 [`metrics.py`](./metrics.py) 中的 `rubric` 字段。

## 3. 评估方法

### 3.1 双模式打分器（`scorer.py`）

- **Mock 模式（默认）**：基于规则的确定性打分。围绕人工标注暴露的两类核心失败模式设计：
  - **"甩锅"话术**：`请联系客服` / `联系快递员` / `查看商品详情页` / `建议耐心等待` 等，把本应自动回复承担的事推给用户或人工；
  - **答非所问**：用户已表达"搞不懂流程"，回复却复述同一套流程（硬封顶有用性 1.8）；
  - **忽略上下文**：用户指代"这个/那款"具体商品，回复却套通用模板；
  - **用户不满时谈内部事项**：如"加强培训/反馈品控"而非先解决问题。
  规则对"主动帮办"（`我可以帮您推荐/查询`）给予加分，并对"主动帮办 + 顺带建议自查"的组合从轻处罚（此时自查属合理补充而非甩锅）。
- **LLM 模式（可选）**：调用 OpenAI 兼容接口（`/v1/chat/completions`）做语义级打分，已内置 DeepSeek / 硅基流动 / OpenAI 支持。环境变量：
  ```bash
  # DeepSeek
  set OPENAI_API_KEY=sk-xxx
  set OPENAI_BASE_URL=https://api.deepseek.com/v1
  set EVAL_LLM_MODEL=deepseek-chat

  # 或：硅基流动 SiliconFlow
  set SILICONFLOW_API_KEY=sk-xxx
  set SILICONFLOW_BASE_URL=https://api.siliconflow.cn/v1
  set EVAL_LLM_MODEL=deepseek-ai/DeepSeek-V3

  python run_eval.py --mode llm
  ```

  **或者把配置写在项目根目录的 `.env` 文件里**（推荐，避免每次开终端都 set）：
  ```bash
  SILICONFLOW_API_KEY=sk-xxx
  SILICONFLOW_BASE_URL=https://api.siliconflow.cn/v1
  EVAL_LLM_MODEL=deepseek-ai/DeepSeek-V3
  ```
  代码启动时会自动加载 `.env`，且环境变量优先级高于 `.env`。

  未配置 key 时自动回退到 mock；调用异常时单条回退到 mock（容错）。
  支持任意 OpenAI 兼容接口，优先读取 `OPENAI_*`，其次 `SILICONFLOW_*`，其次 `DEEPSEEK_*`。

### 3.2 流水线（`run_eval.py`）

1. 读取 20 条自动回复与 20 条人工标注；
2. 逐条调用打分器得到 4 个指标分 + 命中信号（用于可解释性）；
3. 计算加权总分、各指标统计（均值/最小/最大/标准差）、等级分布；
4. 输出最差 3 条（结合人工参考回复与标注做归因分析）与最好 3 条；
5. 写入 `eval_report.md`。

### 3.3 用人工标注校验

`task3_human_ref.json` 的标注一致指出：自动回复普遍"准确但不够有用"，核心问题是把责任推给用户。本方案的 mock 打分结果与之吻合——**准确性均分最高（4.15），有用性均分最低（3.43）**，最差 3 条（case_20 答非所问 / case_13 敏感肌推详情页 / case_01 推责任给用户）均为标注中明确点名的失败 case。

## 4. 评估结论摘要

- **整体加权均分 3.71 / 5.0（良好）**，20 条中 0 条优秀、13 条良好、7 条及格，无严重错误（无瞎编）。
- **最大短板：有用性**（均分 3.43，标准差 0.87 最大，分化严重）。约半数回复以"请联系客服 / 查看商品详情页"收尾，把解决责任甩回用户。
- **准确性普遍良好**（均分 4.15），未发现臆造。
- **最差 3 条**：case_20（用户说搞不懂流程却复述流程，答非所问）、case_13（敏感肌用户被推去看详情页）、case_01（让用户自己联系快递员/客服，无主动服务）。

完整报告见 [`eval_report.md`](./eval_report.md)。

## 5. 局限性讨论

### 5.1 可能评不准的 case

1. **事实准确性无法真正核实**：mock 只能据"是否给出可核对信息"粗判，无法验证退款时效、质保期、民航规定等是否正确。涉及具体政策/规格需要对接知识库或订单系统才能判准。
2. **有用性依赖话术关键词**：`请联系客服`在部分场景下是合理兜底（如超时未到账），会被误罚；规则无法区分"条件性兜底"与"无条件甩锅"。
3. **针对性不懂真正语义**：用户指代"这个/那款"时，规则不知道系统是否真能查到该商品参数——自动回复未查商品有时是系统能力所限，而非回复缺陷，会被错罚。
4. **语气共情对隐性不满不敏感**：如"上次买坏的这次又坏"这类隐性投诉，关键词匹配可能漏判情绪。
5. **样本量小（20 条）**：分布结论不具备统计显著性，仅作定性参考。

### 5.2 改进方向

- **接入真实 LLM 评测**（已支持 `--mode llm`），并以人工标注的 20 条作为一致性校验集，计算与人工排序的 Spearman 相关性。
- **事实维度对接知识库**：对回复中的时效/阈值/政策类陈述与业务知识库自动比对。
- **引入用户反馈信号**：回复后的转人工率、追问率、点赞/点踩作为外部校准。
- **扩充标注集**：覆盖更多品类与情绪场景，定期重算指标分布监控漂移。
- **用人工分数校准规则权重**：回归拟合关键词权重，降低误判。

## 6. AI 工具使用情况

- **代码开发**：使用 Trae IDE（GLM 模型）辅助生成打分器规则、流水线骨架与报告模板。所有代码经人工审查、调参与验证。
- **指标设计**：AI 辅助把业务方模糊诉求拆解为 4 个可量化指标；指标权重与优先级由人工基于 `human_ref` 标注分析确定（核心依据：标注一致指向"有用性"为最大短板）。
- **结果校验**：mock 打分结果与人工标注的吻合度由人工逐条核对（最差 3 条均为标注明确点名的失败 case）。
- **未使用真实 LLM API 跑分**：本交付默认 mock 模式（零成本、确定性）；LLM 模式代码已实现，配置 key 后可直接运行。
